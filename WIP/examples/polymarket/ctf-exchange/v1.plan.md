## Where step 1 was paused

Stopped while inspecting Foundry's `BaseExchangeTest.sol` to mirror its helper
signature in Python. Key reference points already collected:

- `BaseExchangeTest.sol:55-79` — `setUp`: deploys USDC, CTF, ERC1271Mock,
  CTFExchange; calls `registerToken`; `addOperator(bob)` & `addOperator(carla)`.
- `BaseExchangeTest.sol:81-88` — `_prepareCondition` / `_getPositionId` derive
  the `yes` / `no` outcome token ids from `(oracle, questionId, indexSet)`.
  Implementation lives in puya-sol's `out/.../CTHelpers.*` (compiled).
- `BaseExchangeTest.sol:90-151` — `_createOrder`, `_createAndSignOrder*`,
  `_signMessage(pk, hash) → r‖s‖v`. EIP-712 hashing already happens via
  `exchange.hashOrder(order)`.
- `BaseExchangeTest.sol:153-167` — `_mintTestTokens(to, spender, amount)`
  splits `amount` of USDC into yes/no positions via `ctf.splitPosition`.

When resuming: do **not** modify `BaseExchangeTest.sol` (audited code). Mirror
its helpers in `test/conftest.py` only.

## Open questions for resume

- Real CTF (ConditionalTokens) is a complex contract. v2 uses a
  `UniversalMock` that pretends success; v1 may do the same OR a real
  ERC1155 mock with balance bookkeeping is needed for the `testFill*`
  balance-delta assertions. Decide before step 4.
- Foundry's `vm.addr(pk)` returns a 20-byte eth-style address; on AVM the
  orchestrator stores 32-byte AVM addresses. Need to confirm the orchestrator
  treats EOA-recovered addresses (20 bytes) consistently with the maker
  field stored in orders. Inspect a successful `_verifyECDSASignature`
  TEAL path before assuming the obvious mapping.
