## Why each cluster fails

### A. 28 errors — orchestrator program-size cap

All 28 errors throw `algosdk HTTP 400 / "approval program too long. max len 8192 bytes"` at orchestrator deploy. Every test that needs the full `CTFExchange` orchestrator is blocked.

Current binary sizes (`out/exchange/CTFExchange/`):

```
10709  CTFExchange/CTFExchange.approval.bin     ← over the 8192 cap by 2517
 8001  CTFExchange__Helper2/Helper2.approval.bin   (close to cap)
 5069  CTFExchange__Helper1/Helper1.approval.bin
```

**Affected tests:** `test_ctfexchange.py` (22), `test_split_smoke.py` (3), `test_delegate_plumbing.py` (3) = 28.

**Fix path:** puya-sol C++ work — extend the splitter to peel more functions, or land the lonely-chunk delegate runtime that recent commits (e.g. `delegate/v2: dance install/call/revert round-trip lands`) have been building toward. `split.json` currently configures only 2 helper buckets and Helper2 is already ~191 bytes from its own 8192 cap, so simply expanding Helper2 will not work — a third helper bucket and/or the lonely-chunk substitution mechanism is required.

### B. 18 xfails — `test_match_orders.py` module-xfail

Module-level `pytestmark = pytest.mark.xfail(reason="matchOrders is --force-delegate; lonely-chunk runtime not yet wired")`.

These are **skeletons** — every test body is `pytest.fail(...)` with the Foundry test's docstring above it. So they are 1:1 placeholders waiting for the lonely-chunk delegate runtime (same fix path as A).

**Coverage:** 13 of 33 `MatchOrders.t.sol` tests + 5 of 10 `BalanceDeltas.t.sol` tests are skeleton-translated. **15 MatchOrders + 5 BalanceDeltas Foundry tests have no Python skeleton yet** — they should be added when the rest are unblocked, since that's the moment infrastructure is paid down anyway.

### C. 13 xfails — `test_preapproved.py` module-xfail

Module-level reason: `"Preapproved tests need eth-style EOA signing; pending"`.

All 13 of `Preapproved.t.sol` are translated as skeletons; same shape as v1 — they need a Python signing helper (`eth_account.Account.sign_message` → `r ‖ s ‖ v`). This is **identical to v1.plan.md step 1** and could share the helper.

### D. 6 xfails — `test_erc1271_signature.py` module-xfail

Module reason: `"ERC1271 signature path needs an ERC1271 mock + eth-style ECDSA signing infrastructure"`.

All 6 of `ERC1271Signature.t.sol` translated as skeletons. Needs the same EOA signing helper as C, **plus** wiring an `ERC1271Mock` deployment into the orchestrator's signature dispatch path. v2 already has `ToggleableERC1271Mock.sol` and `UniversalMock.sol` available.

### E. 5 xfails — `test_admin.py` Solady ownership

Individual `@pytest.mark.xfail(reason=SOLADY_OWNERSHIP_BROKEN)` on:
`test_initial_owner_is_admin`, `test_transfer_ownership`, `test_add_admin_grants_role`, `test_pause_unpause_flow`, `test_permissioned_ramp_witness`.

The xfail comment blames Solady's `shr(96, shl(96, x))` address-cleanup pattern, but per `PUYA_BLOCKERS.md §1` the actual root cause is **puya backend (Python)**: ABI router codegen consumes the first arg's bytes via `len` without `dup`. The xfail comments are misdiagnosed.

**Fix path:** puya backend Python work — `puya-sol/puya/src/`, not puya-sol C++.

### F. 1 xfail — `test_ecdsa_recover.py`

Reason: `"puya-sol can't translate solady ECDSA's staticcall(gas, 1, ...) ecrecover precompile call to AVM's ecdsa_pk_recover opcode"`.

Per `PUYA_BLOCKERS.md §2` this is actually the Yul `for { let i := 1 } i { i := 0 } { switch ... }` body collapsing to `err` in puya's optimizer — also a **puya backend (Python)** bug.

The **production code** routes around this: `src/exchange/mixins/Signatures.sol` hand-parses `r/s/v` and calls the `ecrecover` precompile directly (workaround already in tree as `// AVM-PORT-ADAPTATION`). Only the direct `ECDSA.recover` probe still fails.

## What's deployable today (revised)

All standalone v2 contracts fit under the 8192-byte cap:

```
1828  CollateralOfframp        ✓ deploys + functional surface
1829  CollateralOnramp         ✓ deploys + functional surface
3382  PermissionedRamp         ✓ deploys + functional surface (witness path needs EOA signing)
4998  CollateralToken          ✗ ctor reverts: `__storage_read/write` takes uint64
                                  but Solidity slot keys are 32-byte bytes (puya bug)
5962  CtfCollateralAdapter     ✓ deploys + functional surface
7011  NegRiskCtfCollateralAdapter ✗ approve() inner-call selector mismatch
                                    (EVM keccak vs ARC4 sha512_256)
 549  UniversalMock            ✓
```

So **4 of 6 standalone contracts deploy and respond to ABI calls**. The 2
that don't are blocked by separate puya issues that should land in their
own task tickets (smaller in scope than the orch's lonely-chunk work):

- **CollateralToken status (3 bugs nested)** — patches landed this branch:

  1. **Slot type mismatch — FIXED in puya-sol C++.**
     `src/builder/assembly/BitwiseShiftOps.cpp::handleSload` and
     `SignedOps.cpp::handleSstore` only converted `biguint → uint64` for
     the slot arg. Solady's `Initializable.initializer()`/`_disableInitializers()`
     do `bytes32 s = _initializableSlot()` then `sload(s)/sstore(s, …)` —
     the Solidity bytes32 type carried into Yul as `BytesWType`, so the
     biguint-only branch missed it and `__storage_read/write` received bytes
     for a slot arg the routine treats as uint64. Patch: extend the same
     `safeBtoi`-via-reinterpret-to-biguint path to `BytesWType`. Verified at
     AWST: 73 / 73 `__storage_*` calls now have `wtype=uint64` (was
     64 + 9 `bytes[32]`).

  2. **Splitter orch picker — FIXED in puya-sol C++.**
     `src/main.cpp:970-986` picked the *first* non-helper contract as the orch.
     For `contract CollateralToken is UUPSUpgradeable, …` the AST resolution
     puts solady's `CallContextChecker` (UUPS parent) before `CollateralToken`.
     So the emitted orch was a 104-byte stub whose ARC-56 had a single method
     `__delegate_update` instead of the 39-method CollateralToken surface.
     Patch: walk to the END and keep the last non-helper contract — that's
     the most-derived concrete contract from the .sol file. After the patch
     `out/collateral/CollateralToken/CollateralToken/CollateralToken.{teal,bin,arc56}`
     re-appears at 4933 bytes with all 39 methods.

  3. **Constructor `box_create` MBR — NOT FIXED (needs puya backend).**
     Once the orch is correctly emitted and slot conversion works, deploy
     fails at the create-txn level: `account … balance 0 below min 3384500`.
     Cause: the constructor calls `_disableInitializers()` which writes
     `_INITIALIZABLE_SLOT` via puya's `__storage_write`, and the first call
     to that routine does `box_create __dyn_storage 8192` synchronously.
     The 8KB box's MBR (~3.4M µAlgos) is checked at end-of-txn against the
     newly-created app account, which has 0 balance because we haven't
     funded it yet. Cannot fund pre-create without predicting the future
     `app_id` (no API). For non-UUPS contracts (`CollateralOnramp` etc.)
     puya defers the box-creating writes to a `__postInit` method, allowing
     the standard "create → fund → __postInit" sequence to work.

     **Fix needed (out of puya-sol C++ scope, lives in puya backend Python):**
     defer the first `box_create` from `__storage_read/write` to a
     post-create init method (e.g., `__postInit`) for UUPS proxies too —
     OR generate a separate "boxes pre-allocated" code path that the
     constructor avoids. Until that lands, `test_collateral_token_deploys`
     stays `@pytest.mark.skip` with the reason updated to point at this
     specific issue (was "puya-sol bug: % opcode arg type coercion" —
     that's actually fixed now).

  See `MEMORY.md → v2_collateral_token_analysis.md` for the trace, fix
  patches, and validation steps.
- **NegRiskAdapter selector mismatch**: the constructor calls `IERC20(token)
  .approve(...)` and the inner-call's method selector is `0x095ea7b3`
  (EVM-keccak of `"approve(address,uint256)"`) — but puya emits selectors
  via ARC4 sha512_256, so the helper's router can't dispatch to it.
  Likely fix: align selector hashing across the contract→helper boundary.
  Pre-existing v2 work touches this; PUYA_BLOCKERS.md covers some of it.

   - `CollateralToken.t.sol` (32) — deploys CollateralToken UUPS proxy + USDC mock; tests wrap/unwrap, mint/burn, decimals/symbol/name, role grants, upgradeToAndCall.

   - `CtfCollateralAdapter.t.sol` (8) and `NegRiskCtfCollateralAdapter.t.sol` (12) — split/merge/redeem/convertPositions tests.
3. **Translate adapter Match\* tests** (16 total) — uses orch BUT… these may not need the full orchestrator — confirm. If they do, they get blocked behind Tier 2.

### Tier 2 — puya-sol C++ (orchestrator size)

4. **Shrink the orch under 8192.** Either:

   The moment the orch deploys: 22 `test_ctfexchange.py` tests + 3 `test_split_smoke.py` + 3 `test_delegate_plumbing.py` go live. Of those, the 22 should immediately pass since the test scenarios don't trigger the documented backend bugs.

5. **Make 18 `test_match_orders.py` tests live.** Same lonely-chunk fix as 4. Once orch deploys + delegate runs, drop the module xfail; some tests may surface secondary bugs.

### Tier 3 — puya backend Python (out of `puya-sol/src/` scope)

6. **PUYA_BLOCKERS #1 (ABI router arg bytes drop)** — unblocks 5 `test_admin.py` xfails.
7. **PUYA_BLOCKERS #2 (ECDSA `for { 1 } switch` collapse)** — unblocks `test_ecdsa_recover.py`. (Production path is workaround'd already.)

## Open questions

- **Adapter Match\* tests** — unclear whether they require the orchestrator or test the adapters in isolation. Audit before deciding Tier 1 vs Tier 2.
- **Solady ownership xfails** — confirm they truly resolve once PUYA #1 is fixed in puya backend, or if a separate puya-sol-side workaround (a peephole on the Solady patterns?) would be lower-effort.
- **Lonely-chunk runtime status** — recent commits show round-trip working (`delegate/v2: dance install/call/revert round-trip lands`) but the `--force-delegate` path that swaps orch's approval mid-call isn't fully wired. Need a status read on what's left.
- **EIP-712 domain hash compatibility** — when EOA-signed orders cross from Python (eth-style) to AVM, the domain separator must agree. Decide: (a) compute the EIP-712 domain in Python the same way the orch does, or (b) call `exchange.hashOrder(order)` from a view method to get the canonical hash, then sign that. Option (b) avoids divergence.
